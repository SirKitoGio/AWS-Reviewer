Introduction  
1.1 Purpose of the Exam  
The AWS Certified Cloud Practitioner (CLF-C02) is the foundational-level  
certification offered by Amazon Web Services. It is designed to validate a candidate’s  
overall understanding of the AWS Cloud — its concepts, services, pricing, and  
support models — without requiring hands-on technical experience.  
This exam serves several important purposes:  
• It establishes a common language around cloud computing for individuals  
across all business functions — not just engineers or developers.  
• It is the recommended entry point into the AWS certification path for those  
new to cloud computing.  
• It demonstrates that a candidate understands why cloud computing matters and  
how AWS delivers value, even if they do not yet configure services themselves.  
Whether you are a developer, project manager, sales professional, finance analyst, or  
executive, this certification proves you understand how your organisation operates in  
the cloud.  
1.2 Tasks the Exam Validates  
The CLF-C02 exam validates that a candidate can:  
• Explain the value proposition of the AWS Cloud.  
• Understand and apply the AWS Shared Responsibility Model.  
• Describe the security best practices for the AWS Cloud.  
• Understand AWS Cloud compliance concepts.  
• Identify core AWS services across compute, storage, networking, and  
databases.

• Identify billing, pricing, and support models available on AWS.  
• Describe common cloud migration strategies.  
• Explain the basics of cloud economics and the financial benefits of the cloud.  
1.3 Target Candidate Description  
The ideal candidate for this exam is someone who:  
• Has 6 months or less of experience with AWS (or is brand new to it).  
• Works in a non-technical business role (e.g., finance, sales, marketing, legal)  
and needs to understand the cloud at a strategic level.  
• Is a technical beginner seeking to validate foundational cloud knowledge before  
pursuing more advanced certifications.  
• Is a student or career-changer entering the cloud field.  
Exam Tip  
You do not need to know how to configure AWS services to pass this exam. The  
focus is on understanding what services do, when to use them, and how  
they relate to business value. Think conceptually, not operationally.  
1.4 Recommended AWS Knowledge Areas  
AWS recommends that candidates have a basic understanding of the following before  
sitting the exam:  
• IT fundamentals: Networking basics (what an IP address is, what a firewall  
does), storage types, and compute concepts.  
• Cloud computing concepts: What cloud computing is and how it differs from  
traditional on-premises computing.  
• Basic AWS familiarity: What AWS is and that it offers cloud services such as  
virtual machines, databases, and storage.  
There are no formal prerequisites to register. However, working through the free AWS  
Cloud Practitioner Essentials course before your exam is strongly recommended.  
1.5 Out-of-Scope Job Tasks  
The following are explicitly out of scope for the CLF-C02 exam:  
• Coding or scripting (e.g., writing Lambda functions or CLI scripts).  
• Designing cloud architectures or solutions.

Many candidates over-study technical implementation details. The Cloud Prac-

1.6 Exam Format  
• Question types: Two types of questions appear on the exam:  
– Multiple choice: One correct answer from four options.  
– Multiple response: Two or more correct answers from five or more options.  
The question will always tell you how many answers to select.  
• Total questions: 65 questions per attempt.  
• Duration: 90 minutes.  
• Delivery: Available in-person at a Pearson VUE or PSI test centre, or via online  
proctoring from home or office.  
• Language: Available in multiple languages.  
• Unscored questions: Of the 65 questions, 15 are unscored (used by AWS to  
validate future exam questions). You will not know which ones they are, so treat  
every question as if it counts.  
1.7 Passing Score and Scaled Scoring Model  
• Passing score: 700 out of 1000\.  
• Scaled scoring: AWS uses a scaled scoring model. This means each question is  
not worth exactly the same number of points. Harder questions may be weighted  
slightly differently. The score does not simply represent the raw percentage  
correct.  
• No penalty for guessing: There is no negative marking. Always answer every  
question — even if you are unsure.  
Exam Tip  
A score of 700/1000 roughly corresponds to answering about 70% of scored  
questions correctly. Aim for 80%+ confidence across all domains to pass  
comfortably.

2.1.1 The Value Proposition of the AWS Cloud  
Before examining specific benefits, it is important to understand what cloud comput-  
ing is. The National Institute of Standards and Technology (NIST) defines cloud  
computing as:  
Definition  
Cloud computing is a model for enabling ubiquitous, convenient, on-demand  
network access to a shared pool of configurable computing resources (e.g.,  
networks, servers, storage, applications, and services) that can be rapidly  
provisioned and released with minimal management effort or service provider  
interaction.  
Inpracticalterms, cloudcomputingmeansrentingcomputingpowerandstorage  
from a provider rather than buying, owning, and operating your own servers. AWS  
is the world’s largest and most comprehensive cloud platform.

2.1.2 Economies of Scale  
One of the most powerful drivers of the cloud value proposition is economies of  
scale.  
Definition  
Economies of scale occur when the cost per unit decreases as the total volume  
increases. AWSoperatesatmassivescale, servingmillionsofcustomersworldwide.  
Because of this, AWS can purchase hardware, bandwidth, and energy at far  
lower prices than any individual organisation could achieve on its own — and it  
passes those savings on to customers through lower prices.  
What this means for you:  
• As AWS grows, its prices generally decrease (AWS has reduced prices over 100  
times since 2006).  
• Small companies immediately benefit from infrastructure that previously only  
large enterprises could afford.  
• You pay only for what you use — there is no need to purchase excess capacity  
“just in case.”  
Exam Tip  
When the exam asks about cost savings from moving to the cloud, look for  
answers that mention economies of scale, pay-as-you-go pricing, or variable  
costs replacing fixed/capital costs.  
2.1.3 Benefits of Global Infrastructure  
Speed of Deployment  
In a traditional on-premises environment, provisioning a server could take weeks or  
months: purchasing the hardware, shipping it, racking and stacking it, configuring  
the network, and installing software. On AWS, you can launch a server in minutes

using the AWS Management Console, CLI, or an API call.  
This speed fundamentally changes how organisations operate:  
• Faster time to market: New products and features can be deployed rapidly.  
• Rapid experimentation: Teams can spin up test environments, run experi-  
ments, and tear them down without long lead times or sunk costs.  
• Agile development: Development and operations teams can iterate quickly.  
Global Reach  
AWS operates in multiple geographic regions around the world (discussed in  
depth in Domain 3). This means:  
• A startup in Morocco can serve customers in Asia, Europe, and North America  
with low latency by deploying infrastructure close to those customers.  
• A company can achieve data residency complianceby choosing specific regions  
that keep data within a given country or jurisdiction.  
• Disaster recovery solutions that previously required a second data centre can be  
built across multiple AWS Regions.  
Without the cloud, achieving this global footprint would require enormous capital  
expenditure — building or leasing data centres in multiple countries. With AWS, it  
is available on demand.

2.5 Advanced Infrastructure, Design Principles & Service Scopes

In this section, we examine key cloud concepts that form a baseline for advanced architectural decisions, resource limits, and service comparisons tested on the CLF-C02.

2.5.1 Cloud Design Principles & Best Practices
Building and operating solutions in the cloud requires adopting key architectural best practices. These principles ensure scalability, loose coupling, and maximum efficiency:
• Services, Not Servers: Leverage managed and serverless resources (like Amazon Aurora or Amazon DynamoDB) to focus on code and product features instead of racking, stacking, powering, and maintaining physical server infrastructure.
• Think Parallel: Maximize performance by parallelizing requests and computations. A core application of this is the S3 Multipart Upload API, which uploads large objects in parts concurrently, significantly speeding up file transfers.
• Decouple Components: Restructure monolithic applications into loose, asynchronously coupled layers. By leveraging messaging queues like Amazon SQS, components can interact without having dependencies on each other's status or availability.

2.5.2 AWS Service Scopes & Infrastructure Boundaries
AWS resource boundaries define where services operate and how they scale:
• VPC Subnet Boundaries: A subnet resides strictly within a single Availability Zone (AZ). It cannot span multiple AZs. This design safeguards resources from multi-AZ failures.
• Global Services: Services whose control planes span all AWS regions. Examples include IAM, Route 53, Amazon CloudFront, AWS STS (Security Token Service), and AWS Billing Conductor.
• Regional Services: Services that operate within a specific region. Examples include Amazon RDS, Amazon DynamoDB, Amazon EFS, and AWS Batch.
• Zonal Services: Tied to a single Availability Zone. An example is Amazon EBS volumes and EC2 instances. Note that while EBS volumes are zonal, their snapshots are regional.
• Region Selection Criteria: Always evaluate region placement based on Latency (proximity to your end-users) and Data Sovereignty/Compliance (ensuring data resides in specific legal jurisdictions).

2.5.3 Secure Storage Lifespan & Decommissioning
• Secure Decommissioning Standards: AWS is responsible for physical media disposal. When storage devices reach their end-of-life or are decommissioned, AWS uses strict methods aligned with NIST SP 800-88 guidelines to ensure customer data is securely destroyed and cannot be recovered before the media is physically reused or destroyed.

2.5.4 Relational, NoSQL, and Columnar Database Choices
Choosing the correct database model is critical for performance and cost:
• Amazon Aurora: A highly scalable MySQL- and PostgreSQL-compatible relational (OLTP) database. It is designed to automatically replicate data across 3 AZs and features self-healing storage capabilities.
• Amazon DynamoDB: A fully managed NoSQL key-value and JSON document database that provides predictable, single-digit millisecond latency at any scale.
• Amazon Redshift: A columnar storage database optimized for Online Analytical Processing (OLAP), data warehousing, and complex business intelligence queries.

2.5.5 AWS Cloud Adoption Framework (CAF) Core Perspectives
The CAF helps organisations map out digital transformation strategies across six perspectives. The CCP exam frequently tests two key perspectives:
• Business Perspective: Aligns strategic business goals with cloud investments, focusing on strategy, ROI, portfolio management, and business value.
• Security Perspective: Focuses on establishing compliance, IAM identity controls, threat detection, infrastructure protection, data protection, and incident response.

2.5.6 Shared Controls and Infrastructure Patching
• Host vs. Guest Patching: Under the Shared Responsibility model, AWS is responsible for applying security updates and patches to the EC2 hypervisor and physical hosts. The customer, however, is responsible for patching the guest operating system (guest OS) and any applications (e.g., databases) they run on virtual instances.

2.5.7 Amazon Chime and Amazon Connect
• Amazon Chime: A managed enterprise communication and productivity service that provides meetings, video calls, and business chat.
• Amazon Connect: An omnichannel cloud-based contact center service that makes it easy to set up customer help centers, integrate chat/voice channels, and leverage automated routing via services like Amazon Lex.

• A subnet is bound to EXACTLY one AZ. It can never span multiple AZs.

3.3.1 AWS Identity and Access Management (IAM)  
Definition  
AWS IAM (Identity and Access Management) is the service that controls  
who can do what in your AWS account. It allows you to create and manage  
users, groups, and roles, and attach policies that define the permissions each  
identity has.  
IAM is a global service — IAM users and roles work across all AWS Regions.  
There is no additional charge for using IAM.  
3.3.2 The Root User Account  
When you first create an AWS account, you log in with an email address and  
password. This is the root user — the most privileged identity in the account.  
Common Trap / Misconception  
The root user has unrestricted access to everything in the account and  
cannot be restricted by IAM policies. AWS strongly recommends:  
1\. Enable MFA on the root user immediately.  
2\. Do not use the root user for everyday tasks.  
3\. Create individual IAM users with only the permissions they need.  
4\. Store root credentials safely and use them only for tasks that cannot be  
done otherwise.  
Tasks Only the Root User Can Perform  
Certain actions in AWS can only be done by the root user and cannot be delegated:  
• Changing the AWS account name, email address, or root user password.  
• Closing (deleting) the AWS account.  
• Changing the AWS Support plan.  
• Restoring IAM user permissions if the only admin has locked themselves out.  
• Enabling MFA Delete on an S3 bucket.  
• Submitting a reverse DNS request to AWS.  
• Agreeing to the GovCloud terms of service.

Exam Tip

Tasks like creating users, launching EC2 instances, or configuring S3 — these

3.3.3 IAM Core Components  
Users  
An IAM user represents a person or application that interacts with AWS. Each  
user has:  
• Console password: For accessing the AWS Management Console via browser.  
• Access keys: For programmatic access (CLI, SDK, API). Composed of an Access  
Key ID and Secret Access Key.  
Best practice: Create one IAM user per person. Never share credentials.  
Groups  
An IAM group is a collection of IAM users. You attach policies to groups, and all  
users in the group inherit those permissions. This simplifies permission management:  
• Instead of attaching 20 policies to 50 individual users, create a “Developers”  
group with the appropriate policies, then add all developers to that group.  
Common Trap / Misconception  
IAM groups cannot contain other groups. They can only contain IAM users.  
Roles  
An IAM role is similar to a user in that it has permissions attached via policies,  
but:  
• A role is not associated with a specific person. It is meant to be assumed  
temporarily by trusted entities.  
• Entities that can assume a role include: EC2 instances, Lambda functions, other  
AWS accounts, federated users (via SAML or OIDC).  
• Roles do not have permanent credentials — when assumed, they provide tempo-  
rary security credentials.  
Common role use cases:  
• An EC2 instance assumes a role to call the S3 API (instead of embedding access  
keys in the application code).  
• A Lambda function assumes a role to write to a DynamoDB table.

• Cross-account roles: Allow a user in Account A to assume a role in Account B  
to access resources there.  
Policies  
An IAM policy is a JSON document that defines permissions. It specifies:  
• Effect: Allow or Deny.  
• Action: WhichAPIactionsareallowed(e.g.,s3:GetObject,ec2:StartInstances).  
• Resource: Which specific resources the actions apply to (e.g., a specific S3  
bucket ARN).  
Two types of policies:  
• AWS Managed Policies: Pre-built policies created and maintained by AWS  
(e.g., AmazonS3ReadOnlyAccess). Recommended for common use cases.  
• Customer Managed Policies: Custom policies you create and manage for  
your specific requirements.  
• Inline Policies: Policies embedded directly into a single user, group, or role.  
Not recommended for general use.  
3.3.4 The Principle of Least Privilege  
Definition  
The principle of least privilege states that every user, system, or process  
should be granted only the minimum permissions necessary to perform  
its job function — and nothing more. This limits the potential blast radius if  
credentials are compromised.  
In practice:  
• AdeveloperwhoonlyneedstoreadfromS3shouldhaves3:GetObject permission  
— not full admin rights.  
• An application that writes to DynamoDB should have only dynamodb:PutItem  
and dynamodb:UpdateItem permissions — not dynamodb:DeleteTable.  
• Start with zero permissions and add only what is needed.

• Access keys: Long-term credentials for programmatic access. Best practices:  
– Never embed access keys in application code or commit them to a source code  
repository.  
– Rotate access keys regularly.

• Password policies: You can configure an IAM password policy that enforces

3.3.6 Credential Storage  
• AWS Secrets Manager: Stores, rotates, and retrieves secrets such as database  
passwords, API keys, and other credentials. It can automatically rotate secrets

• MFA (Multi-Factor Authentication): Adds a second authentication factor

3.4.1 AWS Security Features and Services  
Security Groups  
Definition  
A security group acts as a virtual firewall at the instance level (e.g., for  
EC2 instances). It controls inbound and outbound traffic using allow rules only  
(there are no explicit deny rules — anything not allowed is implicitly denied).  
Key characteristics:  
• Stateful: If you allow inbound traffic, the response is automatically allowed out  
(without needing a separate outbound rule).  
• Applied at the EC2 instance (or ENI) level.  
• Can reference other security groups as sources/destinations.  
• Changes take effect immediately.  
Network Access Control Lists (Network ACLs)  
Definition  
A Network ACL (NACL) acts as a virtual firewall at the subnet level  
within a VPC. Unlike security groups, NACLs support both allow and deny  
rules and are stateless.  
Key characteristics:  
• Stateless: You must create separate rules for inbound and outbound traffic.  
Return traffic is not automatically allowed.  
• Applied at the subnet level — affects all resources in the subnet.  
• Rules are evaluated in order (lowest number first); processing stops at the first  
matching rule.

4.2.1 AWS Regions  
Definition  
An AWS Regionis a geographically distinct area of the worldwhere AWS  
has deployed its infrastructure. Each Region is a separate geographic location  
consisting of multiple, isolated Availability Zones.  
As of 2024, AWS has 33+ Regions worldwide (e.g., us-east-1 in Northern Virginia,  
eu-west-1 in Ireland, ap-southeast-1 in Singapore).  
Key Region characteristics:  
• Regions are completely isolated from each other. A failure in one Region does  
not affect another.  
• Data does not automatically replicate between Regions — customers control  
where their data is stored.  
• Not all AWS services are available in all Regions. Services roll out to  
Regions gradually.  
• Choosing a Region involves considering: latency to users, compliance/data  
sovereignty, service availability, and pricing (prices differ slightly per Region).  
4.2.2 Availability Zones (AZs)  
Definition  
An Availability Zone (AZ) is one or more discrete data centres within a  
Region, each with redundant power, networking, and connectivity. They are  
physically separated by a meaningful distance (typically tens of miles apart) but  
connected with high-speed, low-latency fibre networking.  
Key AZ characteristics:  
• Each Region has a minimum of 3 Availability Zones (e.g., us-east-1a, us-east-  
1b, us-east-1c).  
• AZs are designed so that a disaster (flood, fire, power outage) affecting  
one AZ does not affect others in the same Region.  
• AZs within a Region are interconnected with high-bandwidth, low-latency  
links (suitable for synchronous replication).  
• Deploying across multiple AZs provides high availability within a Region.

Exam Tip  
“Availability Zones do not share single points of failure” is an explicit exam  
objective. Each AZ has its own power supply, cooling, and physical security. A  
hardware failure in AZ-1a does not cascade to AZ-1b.  
4.2.3 Edge Locations  
Definition  
Edge locations are AWS infrastructure endpoints (smaller than a full AZ)  
deployed in major cities around the world. They are used to cache content  
closer to end users and deliver it with lower latency. AWS has 400+ edge  
locations in 90+ cities.  
Edge locations are used by:  
• Amazon CloudFront: AWS’s Content Delivery Network (CDN). Caches copies  
of content (images, videos, web pages) at edge locations closest to the user. When  
a user requests content, CloudFront serves it from the nearest edge location,  
dramatically reducing latency.  
• AWS Global Accelerator: Routes traffic through the AWS global backbone  
network to improve the performance of internet-facing applications by routing  
through the nearest edge location and avoiding the public internet for as long as  
possible.  
4.2.4 High Availability Using Multiple AZs  
Deploying across multiple AZs is the primary mechanism for achieving high avail-  
ability in AWS:  
• Elastic Load Balancing: Distributes traffic across EC2 instances in multiple  
AZs. If instances in one AZ fail, traffic is automatically routed to the remaining  
AZs.  
• Amazon RDS Multi-AZ: Automatically maintains a synchronous standby  
replica in a different AZ. If the primary fails, automatic failover promotes the  
standby.  
• Amazon S3: By default, S3 stores data redundantly across at least three AZs  
within a Region. 99.999999999% (11 nines) durability.  
4.2.5 When to Use Multiple Regions  
Deploying across multiple Regions (as opposed to multiple AZs within one Region)  
is appropriate for:  
• Disaster recovery: Protect against Region-level failures (extremely rare but  
possible, e.g., a natural disaster affecting an entire geographic area).  
37AWS CLF-C02 Study Guide CHAPTER 4\. CLOUD TECHNOLOGY AND SERVICES  
• Business continuity: Maintain operations even in the event of complete Region  
unavailability.  
• Low latency for global users: Serve users in different continents from the  
nearest Region.  
• Data sovereignty: Store data in a specific geographic jurisdiction to comply  
with laws.  
4.2.6 AWS Local Zones and AWS Wavelength Zones  
• AWS Local Zones: Extensions of AWS Regions to specific metropolitan  
areas (e.g., Los Angeles, Chicago, Dallas). They bring compute, storage, and  
database services closer to large population centres for ultra-low-latency applica-  
tions (e.g., real-time gaming, live video streaming, industrial automation). Not  
every city has a full Region, but a Local Zone brings AWS closer.  
• AWS Wavelength Zones: Deploys AWS compute and storage services within  
telecommunications (5G) providers’ infrastructure. This puts AWS ca-

latency for mobile applications (e.g., autonomous vehicles, AR/VR, connected  
IoT devices).

or thinking about servers. AWS handles the underlying infrastructure — you  
only manage your code and data.  
• AWS Lambda: Run code in response to events without provisioning servers.  
You upload a function (e.g., a Python or Node.js function), configure a trigger  
(e.g., an HTTP request, an S3 upload, a DynamoDB change), and Lambda  
executes your code automatically. You pay only for the milliseconds of compute  
time used.  
Lambda use cases: API backends, event-driven data processing, scheduled  
tasks (like a cron job), file transformations, real-time stream processing.  
• AWS Fargate: A serverless compute engine for containers. Instead of  
managing the EC2 instances that run your containers (as with ECS on EC2),  
Fargate removes the need to provision or manage servers — you just define the  
container and Fargate handles the rest.  
Lambda vs. Fargate vs. EC2  
• Lambda: Short-lived, event-driven functions. No server management. Max  
15 minutes per execution.  
• Fargate: Containerised workloads without managing EC2. Good for longer-  
running processes.  
• EC2: FullcontrolovertheOS,networking, andhardware. Bestforspecialised  
or complex workloads.

Definition  
AWS Auto Scaling automatically adjusts the number of EC2 instances (or

amount of capacity at all times. It provides elasticity — the cloud’s key  
advantage over on-premises.  
Auto Scaling can:  
• Scale out (add more instances) when traffic spikes.  
• Scale in (terminate instances) when traffic drops, reducing costs.  
• Maintain a minimum and maximum number of instances.

4.3.5 Load Balancers  
Definition  
A load balancer distributes incoming application traffic across multiple tar-  
gets (e.g., EC2 instances) in multiple Availability Zones. It improves availability,  
fault tolerance, and scalability.  
AWS offers the Elastic Load Balancing (ELB) service with three types:  
• Application Load Balancer (ALB): Layer 7 (HTTP/HTTPS). Routes based  
on URL paths, hostnames, or HTTP headers. Best for web applications and  
microservices.  
• Network Load Balancer (NLB): Layer 4 (TCP/UDP). Extremely high perfor-  
mance, low latency. Best for real-time applications, gaming, and high throughput.  
• Gateway Load Balancer (GWLB): For deploying and scaling third-party  
virtual network appliances (firewalls, intrusion detection systems).  
Key purpose: Ensure no single instance receives all the traffic (preventing overload)  
and automatically route traffic away from unhealthy instances.

4.5.1 Amazon VPC (Virtual Private Cloud)  
Definition  
An Amazon VPC is a logically isolated section of the AWS Cloud  
that you define. Within your VPC, you control your own IP address range,  
subnets, routing tables, and network gateways. Think of it as your own  
private data centre network, hosted on AWS.  
VPC Key Components  
• Subnets: Subdivisions of the VPC IP address range. A public subnet has a  
route to the internet; a private subnet does not.  
• Internet Gateway (IGW): Allows resources in public subnets to communicate  
with the internet (inbound and outbound).  
• NAT Gateway: Allows resources in private subnets to initiate outbound  
internet connections (e.g., to download updates) without being directly reachable  
from the internet. “NAT” \= Network Address Translation.  
• Route Tables: Define how traffic is directed within the VPC (e.g., send all  
0.0.0.0/0 traffic to the internet gateway).  
• VPC Peering: Connects two VPCs privately (within the same or different  
accounts/regions), allowing resources to communicate as if on the same network.  
• AWS Transit Gateway: A network hub that connects thousands of VPCs  
and on-premises networks in a hub-and-spoke model (simplifies VPC peering at  
scale).

4.5.2 Security Within a VPC  
Two layers of network security within a VPC (already detailed in Domain 2):  
• Security Groups: Stateful, instance-level firewall.  
• Network ACLs: Stateless, subnet-level firewall.  
4.5.3 Amazon Route 53  
Definition  
Amazon Route 53 is AWS’s highly available and scalable DNS (Domain  
Name System) service. It translates human-friendly domain names (e.g.,  
www.example.com) into IP addresses. Route 53 also provides domain registration  
services.  
Key Route 53 features:  
• DNS routing policies: Simple, Weighted, Latency-Based, Failover, Geolocation,  
Geoproximity, Multivalue.  
• Health checks: Route 53 monitors the health of your endpoints and routes  
traffic away from unhealthy ones.  
• Failover routing: Automatically redirect traffic to a standby site if the primary  
fails.  
4.5.4 Edge Services  
• Amazon CloudFront: A Content Delivery Network (CDN) that caches  
content at 400+ global edge locations, reducing latency for end users. Integrates  
with S3, EC2, ALB, and Lambda@Edge. Supports HTTPS and integrates with  
AWS WAF and Shield.  
• AWS Global Accelerator: Routes user traffic through the AWS global  
backbone network (not the public internet) from the nearest edge location  
to your application, improving performance and availability. Unlike CloudFront  
(which caches content), Global Accelerator is for non-cacheable, dynamic  
traffic such as gaming or financial transactions.  
CloudFront vs. Global Accelerator  
• CloudFront: Best for cacheable content(images, videos, static web pages).  
Reduces latency by serving from cache.  
• Global Accelerator: Best for dynamic, non-cacheable traffic. Improves  
performance by routing over AWS’s private network, not the internet.

5.1.1 EC2 Purchasing Options  
One of the most important areas for the exam. AWS offers multiple ways to pay for

On-Demand Instances  
Definition  
On-Demand Instances allow you to pay for compute capacity by the hour or  
second (depending on the OS) with no upfront commitment or long-term  
contract. You start and stop paying the moment you launch or terminate the  
instance.  
• Best for: Irregular, short-term, or unpredictable workloads; new applications  
being tested; development and testing environments.  
• Cost: Highest per-unit cost of all EC2 options. The “list price” baseline.  
• Commitment: None. Start and stop any time.

Reserved Instances (RIs)  
Definition  
Reserved Instances provide a significant discount (up to 72%) compared  
to On-Demand pricing, in exchange for a 1-year or 3-year commitment to  
use that instance type.  
• Best for: Steady-state, predictable workloads where you know you’ll run the  
instance for at least a year (e.g., a production web server, a database server).  
• Payment options: All Upfront (maximum discount), Partial Upfront, No  
Upfront (minimum discount).  
• Convertible RIs: Allow you to change the instance family, OS, or tenancy  
during the term (slightly lower discount than standard RIs).  
• Standard RIs: Fixed instance type; cannot be changed. Higher discount.  
Reserved Instances in AWS Organizations: If you have multiple accounts under  
AWS Organizations, reserved instance discounts can be shared across all accounts  
in the organisation (via consolidated billing), maximising discount utilisation even if  
one account underuses its RIs.  
Savings Plans  
Definition  
Savings Plans offer flexible discounts (up to 72%) over On-Demand, in  
exchange for a commitment to a minimum spend per hour (e.g., $10/hour)  
for 1 or 3 years. Unlike RIs, Savings Plans are not tied to a specific instance  
type.  
Two types:  
• Compute Savings Plans: Apply to any EC2 instance regardless of family, size,  
AZ, Region, OS, or tenancy. Also apply to AWS Fargate and Lambda. Most  
flexible.  
• EC2 Instance Savings Plans: Commit to a specific instance family in a Region.  
Higher discount (up to 72%) but less flexible.  
Reserved Instances vs. Savings Plans  
• RIs: Commit to a specific instance type (size, region, OS). Less flexible but  
simple.  
• Savings Plans: Commit to a spend rate. More flexible, applies to any EC2  
\+ Fargate \+ Lambda.  
Savings Plans are generally the newer, more recommended approach for most  
organisations.

Spot Instances  
Definition  
Spot Instances let you use spare AWS EC2 capacity at discounts of up  
to 90% compared to On-Demand. However, AWS can reclaim (interrupt)  
the instance with 2 minutes notice when it needs the capacity back.  
• Best for: Fault-tolerant, stateless, or batch workloads that can be interrupted  
and resumed — data processing, video rendering, scientific simulations, ML  
training (checkpointing), CI/CD pipelines.  
• NOT suitable for: Databases, production web servers, any workload that  
cannot tolerate interruption.  
Dedicated Hosts  
Definition  
A Dedicated Host is a physical EC2 server dedicated entirely to your use —  
no other AWS customers share the physical hardware. You have visibility into  
the underlying physical infrastructure (sockets, cores) and can use your existing  
server-bound licences (BYOL).  
• Best for: Compliance requirements that prohibit multi-tenancy (e.g., certain  
government or financial regulations); Windows Server, SQL Server, RHEL, or  
SUSE licences tied to physical sockets/cores.  
• Most expensive option.  
Dedicated Instances  
• Run on hardware dedicated to a single customer (similar to Dedicated Hosts),  
but you do not have visibility or control over the physical host placement.  
• Less flexible than Dedicated Hosts for licence management but cheaper.  
Dedicated Hosts vs. Dedicated Instances  
• Dedicated Host: You control placement on a specific physical server. Full  
socket/core visibility for BYOL compliance. Higher cost.  
• Dedicated Instance: Hardware dedicated to your account but placement  
managed by AWS. Simpler. Slightly cheaper.

5.1.3 Data Transfer Charges  
AWS data transfer pricing can be a source of surprise costs. Key rules:  
• Data IN to AWS: Almost always free (ingress is free).  
• Data OUT from AWS to the internet: Charged per GB. First 100 GB/month  
is free; beyond that, rates apply (varies by Region, typically $0.09/GB from  
us-east-1).  
• Data transfer between AZs within the same Region: Small charge per  
GB in each direction.  
• Data transfer within the same AZ: Free (if using private IP).  
• Data transfer between Regions: Charged (higher than intra-Region).  
• Using CloudFront: Can reduce data transfer costs because CloudFront’s egress  
pricing is lower than direct S3/EC2 egress, and edge caching reduces the number  
of origin fetches.  
Exam Tip  
Remember: incoming data to AWS is free. Outgoing data to the internet  
costs money. Data between services within the same AZ on private IP is  
free.

Storage pricing varies by:

• Storage class: S3 Standard is more expensive per GB than S3 Glacier; however,  
retrieval from Glacier has additional fees.  
• Requests: S3 charges for API requests (PUT, GET, LIST). More requests \=  
more cost.  
• Data transfer out: Charged when data leaves S3 to the internet.

• A — Patching the hypervisor: AWS responsibility. Customers never access  
the hypervisor.  
• B — Physical data centre security: AWS responsibility. Customers have no

Correct: A — Deploy EC2 instances across multiple Availability Zones.  
High availability means eliminating single points of failure. By distributing instances
